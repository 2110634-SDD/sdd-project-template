# เกณฑ์การประเมินผลโครงงาน (Project Assessment Rubrics)

โครงงานนี้มีน้ำหนักคะแนนรวม **40%** ของทั้งรายวิชา  
โดยแบ่งการประเมินตาม Milestone และมีการใช้ทั้งการตรวจแบบอัตโนมัติ (autograding)
และการประเมินโดยผู้สอน

---

## หลักการให้คะแนนโดยรวม

- คะแนนในแต่ละหัวข้อย่อยจะให้ในช่วง **0–10 คะแนน**
- คะแนนดังกล่าวจะถูกนำไปถ่วงน้ำหนักตามสัดส่วน (%) ของหัวข้อนั้น
- การประเมินมุ่งเน้นทั้ง:
  - ความครบถ้วนและความถูกต้องตามข้อกำหนด
  - คุณภาพของการออกแบบ การตัดสินใจเชิงวิศวกรรม และเหตุผลประกอบ

---

## การใช้ Autograding

ในแต่ละ Milestone จะมีการใช้ **autograding** เพื่อช่วยตรวจสอบ
ความครบถ้วนและความสอดคล้องกับข้อกำหนดที่ระบุไว้ เช่น

- โครงสร้าง repository
- การมีอยู่ของไฟล์ที่จำเป็น
- รูปแบบและ naming convention ของเอกสาร
- การ build / run / test (ใน Milestone ที่เกี่ยวข้องกับการพัฒนา)

อย่างไรก็ตาม  
**autograding จะไม่ใช้ในการตัดสินคุณภาพของการออกแบบ**
หรือความเหมาะสมของการตัดสินใจเชิงสถาปัตยกรรม

สัดส่วนโดยประมาณของคะแนนจาก autograding ในแต่ละ Milestone คือ:

- **Milestone 1:** ประมาณ 40% ของคะแนน Milestone
- **Milestone 2:** ประมาณ 50% ของคะแนน Milestone
- **Milestone 3:** ประมาณ 30% ของคะแนน Milestone

รายละเอียดของสิ่งที่ตรวจด้วยระบบอัตโนมัติ
จะระบุไว้ในไฟล์ `INSTRUCTIONS.md` ของแต่ละ Milestone

---

---

## Milestone 0: Team Setup & Tool Familiarization (Not Graded)

**เป้าหมาย:**  
ให้นิสิตคุ้นเคยกับการทำงานเป็นทีม การใช้ Git/GitHub
และกระบวนการส่งงานก่อนเริ่มโครงงานจริง

### ลักษณะการประเมิน
- ใช้การตรวจแบบอัตโนมัติ (autograding) **100%**
- ตรวจเฉพาะความครบถ้วนและรูปแบบของไฟล์ `TEAM.md`
- ตรวจสอบการมีส่วนร่วมของสมาชิกในทีมผ่าน commit history

> Milestone นี้ **ไม่นำมาคิดคะแนน**
> แต่เป็นส่วนสำคัญในการฝึกกระบวนการทำงาน
> และใช้ทดสอบระบบ autograding ก่อน Milestone อื่น ๆ

---

## **Milestone 1: Foundations & High-Level Design (10%)**
**เป้าหมาย:** วัดความเข้าใจในปัญหา การวิเคราะห์ความต้องการ
และการออกแบบระดับแนวคิด

### 1. คุณภาพของการวิเคราะห์ความต้องการ (40%)
- **ยอดเยี่ยม (9–10):**  
  Use Cases มีคุณค่าต่อ Actor อย่างชัดเจน ไม่ใช่เพียงการระบุฟังก์ชัน  
  Description แบบ Fully Dressed ครอบคลุม Main Flow และ Alternative Flows อย่างเหมาะสม
- **ดี (7–8):**  
  Use Cases ส่วนใหญ่ถูกต้อง Main Flow ชัดเจน  
  แต่ Alternative Flows อาจขาดรายละเอียดบางส่วน
- **พอใช้ (5–6):**  
  ระบุ Use Cases ได้ แต่ยังสับสนระหว่าง Use Case กับฟังก์ชัน  
  Description มีเพียง Main Flow
- **ต้องปรับปรุง (0–4):**  
  ระบุ Use Cases ผิดพลาด หรือไม่มี Description ที่มีความหมาย

### 2. คุณภาพของการออกแบบระดับแนวคิด (40%)
- **ยอดเยี่ยม (9–10):**  
  Conceptual Model (S.O.M.) สะท้อน Domain ได้ถูกต้องและสอดคล้องกับ Use Cases  
  Quality Attribute Scenarios มีความเฉพาะเจาะจงและสามารถวัดผลได้
- **ดี (7–8):**  
  Conceptual Model ส่วนใหญ่ถูกต้อง  
  Quality Attributes ที่ระบุยังค่อนข้างกว้าง
- **พอใช้ (5–6):**  
  มี Conceptual Model แต่ยังขาดความสมบูรณ์หรือมีข้อผิดพลาดสำคัญ
- **ต้องปรับปรุง (0–4):**  
  ไม่มี Conceptual Model หรือไม่สอดคล้องกับ Use Cases

### 3. การนำเสนอและตอบคำถาม (20%)
- **ยอดเยี่ยม (9–10):**  
  นำเสนอเป็นมืออาชีพ แสดงถึงความเข้าใจร่วมกันของทีม  
  สามารถอธิบายและปกป้องการตัดสินใจได้อย่างมีเหตุผล
- **ดี (7–8):**  
  การนำเสนอชัดเจน สมาชิกส่วนใหญ่สามารถตอบคำถามได้
- **พอใช้ (5–6):**  
  การนำเสนอไม่เป็นระบบ หรือมีเพียงบางคนที่เข้าใจเนื้อหา
- **ต้องปรับปรุง (0–4):**  
  ขาดการเตรียมตัว ไม่สามารถตอบคำถามพื้นฐานได้

---

## **Milestone 2: Architectural Checkpoint (15%)**
**เป้าหมาย:** วัดความสามารถในการพิสูจน์ว่าสถาปัตยกรรมที่เลือก
สามารถทำงานได้จริงผ่าน Prototype

### 1. การทำงานของ Prototype (40%)
- **ยอดเยี่ยม (9–10):**  
  Live Demo ทำงานได้ครบ End-to-End อย่างราบรื่น  
  มี CI Pipeline ที่ build และทดสอบระบบโดยอัตโนมัติ
- **ดี (7–8):**  
  Demo ส่วนใหญ่ทำงานได้ มี CI Pipeline แต่ยังไม่สมบูรณ์
- **พอใช้ (5–6):**  
  Demo ทำงานได้เพียงบางส่วน ไม่มี CI Pipeline ที่ใช้งานได้จริง
- **ต้องปรับปรุง (0–4):**  
  ไม่มี Demo ที่ทำงานได้ หรือระบบล้มเหลวระหว่างการนำเสนอ

### 2. คุณภาพของการออกแบบและโค้ด (40%)
- **ยอดเยี่ยม (9–10):**  
  Detailed Design สอดคล้องกับโค้ด  
  โครงสร้างโค้ดสะอาด มีการใช้ Design Principles อย่างชัดเจน  
  มี Unit Tests ที่มีความหมาย
- **ดี (7–8):**  
  โครงสร้างโค้ดพอใช้ได้ มี Code Smells บางส่วน  
  มี Unit Tests แต่ครอบคลุมไม่ครบ
- **พอใช้ (5–6):**  
  เอกสารออกแบบไม่สอดคล้องกับโค้ด  
  โครงสร้างไม่ดี และไม่มี Unit Tests
- **ต้องปรับปรุง (0–4):**  
  ไม่มีเอกสารออกแบบ หรือโค้ดยุ่งเหยิง (Big Ball of Mud)

### 3. การสาธิตและอธิบายเชิงเทคนิค (20%)
- **ยอดเยี่ยม (9–10):**  
  อธิบายเหตุผลเบื้องหลังการเลือกเทคโนโลยีและโครงสร้างได้อย่างชัดเจน
- **ดี (7–8):**  
  อธิบายการทำงานของระบบได้ แต่ยังขาดเหตุผลเชิงลึก
- **พอใช้ (5–6):**  
  อธิบายได้ในภาพรวม แต่ไม่สามารถตอบคำถามเชิงเทคนิคได้
- **ต้องปรับปรุง (0–4):**  
  ไม่สามารถอธิบายโค้ดหรือการตัดสินใจทางเทคนิคได้

---

## **Milestone 3: Final Product & Project Portfolio (15%)**
**เป้าหมาย:** วัดคุณภาพของผลิตภัณฑ์สุดท้าย
การพัฒนาคุณภาพของระบบตลอดโครงงาน
และการเรียนรู้ของนิสิต

### 1. ความสมบูรณ์และเสถียรภาพของผลิตภัณฑ์ (30%)
- **ยอดเยี่ยม (9–10):**  
  ฟังก์ชันหลักทำงานได้ครบถ้วนและเสถียร  
  ระบบจัดการ Error Cases ได้ดี
- **ดี (7–8):**  
  ฟังก์ชันหลักส่วนใหญ่ทำงานได้ แต่ยังมี Bug บางส่วน
- **พอใช้ (5–6):**  
  มีเพียงบางฟังก์ชันที่ทำงานได้ ระบบยังไม่เสถียร
- **ต้องปรับปรุง (0–4):**  
  ระบบไม่สามารถทำงานได้ตามฟังก์ชันหลัก

### 2. คุณภาพของการออกแบบและโค้ดโดยรวม (50%)
- **ยอดเยี่ยม (9–10):**  
  โค้ดสะท้อนความเข้าใจในหลักการออกแบบอย่างลึกซึ้ง  
  มีการ Refactoring เพื่อปรับปรุงคุณภาพอย่างเห็นได้ชัด  
  เอกสาร `README.md` สมบูรณ์
- **ดี (7–8):**  
  โค้ดมีโครงสร้างที่ดี มี Bad Smells บางส่วน
- **พอใช้ (5–6):**  
  โค้ดทำงานได้แต่ดูแลรักษายาก
- **ต้องปรับปรุง (0–4):**  
  โค้ดยุ่งเหยิง ไม่มีการประยุกต์ใช้หลักการออกแบบ

### 3. การนำเสนอและสรุปการเรียนรู้ (20%)
- **ยอดเยี่ยม (9–10):**  
  การนำเสนอชัดเจนและน่าสนใจ  
  Reflection สะท้อนการคิดวิเคราะห์และการเรียนรู้ของแต่ละบุคคล
- **ดี (7–8):**  
  สรุปภาพรวมได้ดี แต่ Reflection ยังขาดความลึก
- **พอใช้ (5–6):**  
  การนำเสนอเป็นเพียงการ Demo  
  Reflection มีเนื้อหาผิวเผิน
- **ต้องปรับปรุง (0–4):**  
  ไม่มี Reflection หรือไม่มีคุณภาพ
